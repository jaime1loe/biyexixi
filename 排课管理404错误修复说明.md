# 排课管理404错误修复说明

## 问题描述

排课管理页面加载时出现404错误：
```
Failed to load resource: server responded with a status of 404 (Not Found)
GET /api/api/schedules?semester=2024-2025-2 HTTP/1.1" 404 Not Found
GET /api/api/courses HTTP/1.1" 404 Not Found
GET /api/api/classrooms HTTP/1.1" 404 Not Found
```

## 问题原因

### 1. 路径重复
前端API请求的URL被重复添加了`/api`前缀：

**原因分析**：
1. `frontend/src/api/index.ts`设置了`baseURL: '/api'`
2. API调用时传入的路径是`'/schedules'`
3. axios会自动拼接：`baseURL + url = '/api' + '/schedules' = '/api/schedules'`
4. 但后端路由注册时也使用了`prefix="/api"`
5. 导致实际请求路径变成：`/api/schedules`（正确）

**但是**，有些API文件直接使用了`/api`前缀：
```typescript
// ❌ 错误的做法
request.get('/api/schedules')  // 会变成 /api/api/schedules
```

而应该使用：
```typescript
// ✅ 正确的做法
request.get('/schedules')  // 会变成 /api/schedules
```

### 2. 路由注册不一致

后端main.py中的路由注册方式不一致：
- 有些在main.py中设置完整prefix：`prefix="/api/auth"`
- 有些在router文件中设置prefix，main.py只设置`/api`
- `schedule.router`没有设置prefix，main.py设置了`prefix="/api"`

```python
# main.py
app.include_router(auth.router, prefix="/api/auth")          # 完整前缀
app.include_router(evaluations.router, prefix="/api")          # evaluations.py中设置了prefix="/evaluations"
app.include_router(schedule.router, prefix="/api")              # schedule.py中没有设置prefix
```

## 解决方案

### 方案1：统一使用baseURL

修改`frontend/src/api/index.ts`，将`baseURL`设为空字符串，然后在请求拦截器中自动添加`/api`：

**修改前**:
```typescript
const request = axios.create({
  baseURL: '/api',
  timeout: 30000
})
```

**修改后**:
```typescript
const request = axios.create({
  baseURL: '',
  timeout: 30000
})
```

**请求拦截器修改**:
```typescript
request.interceptors.request.use(
  (config) => {
    const token = sessionStorage.getItem('token')
    if (token) {
      config.headers.Authorization = `Bearer ${token}`
    }

    // 确保URL以/api开头
    if (config.url && !config.url.startsWith('/api')) {
      config.url = '/api' + config.url
    }

    return config
  }
)
```

### 方案2：后端路由统一

后端main.py中schedule路由已正确设置：
```python
app.include_router(schedule.router, prefix="/api", tags=["课表"])
```

### 方案3：前端API路径统一

确保所有API调用不包含`/api`前缀：

**schedule.ts** (已修复):
```typescript
// ✅ 正确：不包含/api
export const getSchedules = (params?: {...}) => {
  return request.get('/schedules', { params })
}

// ❌ 错误：包含/api
export const getSchedules = (params?: {...}) => {
  return request.get('/api/schedules', { params })
}
```

## 最佳实践建议

### 1. 统一baseURL设置
- **推荐**：`baseURL: ''` + 拦截器自动添加`/api`
- **优点**：灵活，可以控制哪些API需要`/api`前缀
- **缺点**：需要在拦截器中处理

### 2. 统一API调用规范
- **规则**：API路径不包含`/api`前缀
- **示例**：`'/schedules'` 而不是 `'/api/schedules'`
- **优点**：代码简洁，易于维护

### 3. 统一后端路由注册
- **规则**：所有路由在main.py中使用`prefix="/api"`
- **示例**：`app.include_router(router, prefix="/api")`
- **优点**：路径一致，易于理解
